# Operator 空选择判定回归复盘（2026-02-27）

## 背景
在 Blender 5.0 环境下，多个 Operator 在“没有选中对象/mesh/collection”时出现行为异常。已知表现包括：应当报错并取消执行却继续走流程、无提示静默返回、以及局部场景下可能触发后续索引或状态错误。

## 现象
1. `Isolate Collections` 在空选择时未按预期退出 local view（已先行修复）。
2. 多个 Operator 仍使用 `is None` 判断列表返回值，导致空列表 `[]` 未被识别为“未选择”。
3. 用户侧体验不一致：有的按钮空选中会报错，有的按钮会继续执行但不产生有效结果。

## 根因
### 根因 1：返回契约与调用方判空方式不一致
- `Collection.get_selected()`、`filter_type()`、`Collection.filter_hst_type()` 均以“列表”语义返回结果。
- 无匹配时返回空列表 `[]`，而不是 `None`。
- 调用方仍使用 `if xxx is None:`，导致空列表分支失效。

### 根因 2：历史代码混用两种判空风格
- 同一仓库内同时存在 `if not xxx` 与 `if xxx is None`。
- 缺少统一规范与检查，导致回归长期潜伏，升级 Blender 后被触发并集中暴露。

## 影响范围
本次修复覆盖下列 operator 文件中的指定点位：
- `operators/collection_ops.py`
- `operators/decal_ops.py`
- `operators/collision_ops.py`
- `operators/env_ops.py`
- `operators/cad_ops.py`
- `operators/attribute_ops.py`
- `operators/uv_ops.py`
- `operators/origin_ops.py`

## 修复策略
1. 对由上述列表型 helper 返回的变量，统一将 `if xxx is None:` 改为 `if not xxx:`。
2. 保持原有提示文本、返回码（`{"CANCELLED"}`）和业务流程不变，只修正判空逻辑。
3. 不改 helper 的返回契约，避免扩大回归面。

## 验证结果
1. 代码级验证：对修改文件执行 `python -m py_compile`，语法通过。
2. 行为级预期：空选择触发时，相关 operator 都会正确进入报错/取消路径，不再静默继续执行。
3. 回归重点：`Isolate Collections` 在 local view 下空选择可正确退出隔离。

## 预防措施
1. 统一规范：列表型返回值一律使用 truthy 判空（`if not xxx`）。
2. Code Review 清单新增检查项：调用方判空是否与返回契约一致。
3. 增加回归扫描脚本规则：
   - 搜索 `filter_type(...)`/`Collection.get_selected()`/`Collection.filter_hst_type(...)` 后紧跟 `is None` 的写法并提示审查。
